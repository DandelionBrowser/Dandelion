# Purpose

The difference between a functional browser and a premium browser is polish.

Polish is not decoration.

Polish communicates quality.

Every interaction should feel intentional.

Every animation should have purpose.

Every component should provide feedback.

The browser should feel handcrafted.

---

# Philosophy

Users notice inconsistency long before they notice perfection.

Small improvements accumulate.

Tiny frustrations accumulate.

Every interaction matters.

Design every interaction as though it will be used thousands of times every day.

---

# Quality Standard

Nothing should feel unfinished.

Every screen should answer:

Is this obvious?

Is this responsive?

Is this visually balanced?

Does it communicate state?

Does it feel premium?

---

# Interaction Feedback

Every interaction should acknowledge the user.

Examples:

Hover

Press

Focus

Selection

Drag

Loading

Completion

Failure

Never leave the user wondering whether input was received.

---

# Hover States

Every interactive element should respond immediately.

Hover feedback may include:

Background colour

Elevation

Border colour

Icon colour

Opacity

Cursor

Scale (very subtle)

Avoid dramatic movement.

Hover should feel effortless.

---

# Active States

Buttons should visibly depress.

Tabs should clearly indicate activation.

Menu items should confirm selection.

Interactions should never feel "dead."

---

# Focus States

Keyboard users should always know where focus is.

Focus indicators should:

Be highly visible

Match the design language

Remain consistent

Never remove focus outlines without an equivalent replacement.

---

# Loading States

Never leave empty space while work is occurring.

Prefer:

Skeletons

Progress indicators

Inline loading

Button spinners

Optimistic UI where appropriate

Avoid full-page loading screens whenever possible.

---

# Empty States

Empty screens should provide direction.

Explain:

Why the area is empty

What the user can do

How to begin

Avoid blank containers.

---

# Error States

Errors should be:

Human

Actionable

Recoverable

Explain:

What happened

What can be done

Avoid technical language unless necessary.

---

# Success States

Acknowledge completion subtly.

Examples:

Checkmark

Toast

Animated icon

Button confirmation

Do not overuse celebrations.

---

# Motion Philosophy

Motion should communicate.

Not entertain.

Animations should explain:

Movement

Hierarchy

State changes

Relationships

Never animate purely for decoration.

---

# Motion Timing

Suggested durations:

Micro interactions

100-120ms

Buttons

150-180ms

Menus

180-220ms

Panels

220-280ms

Dialogs

250-320ms

Avoid animations exceeding 400ms.

---

# Motion Curves

Prefer natural easing.

Avoid:

Linear

Bounce

Elastic

Exaggerated overshoot

Motion should feel effortless.

---

# Layout Stability

Prevent layout shifts.

Content should not unexpectedly move.

Reserve space for:

Images

Icons

Loading indicators

Avoid jumping interfaces.

---

# Visual Rhythm

Maintain consistent spacing.

Prefer an 8px spacing system.

Elements should align visually even when not mathematically identical.

Optical alignment is more important than perfect geometry.

---

# Component Consistency

Buttons should behave like every other button.

Inputs should behave like every other input.

Menus should behave like every other menu.

Consistency builds confidence.

---

# Responsive Behaviour

Support:

Small laptop

Large desktop

Ultrawide

High DPI

Window resizing

Collapsed layouts

Interfaces should adapt gracefully.

---

# Scroll Behaviour

Scrolling should remain smooth.

Avoid nested scrolling.

Restore previous positions where expected.

Avoid scroll jumps.

---

# Context Menus

Menus should appear instantly.

Dismiss naturally.

Support:

Mouse

Keyboard

Touchpad

Avoid deep nesting.

---

# Drag & Drop

Dragging should feel physical.

Provide:

Hover targets

Insertion indicators

Smooth movement

Predictable drop zones

Never make drag operations ambiguous.

---

# Browser Tabs

Tabs should:

Animate naturally

Resize smoothly

Clearly indicate active state

Provide subtle hover feedback

Support drag animations

Loading should feel alive without being distracting.

---

# Sidebars

Collapse smoothly.

Remember width.

Animate intelligently.

Avoid snapping.

---

# Tooltips

Only appear when useful.

Delay slightly.

Disappear naturally.

Never obstruct important content.

---

# Toast Notifications

Toasts should:

Appear consistently

Avoid blocking interaction

Dismiss automatically

Support manual dismissal

Avoid stacking excessively.

---

# Dialogs

Dialogs should:

Open smoothly

Trap focus

Support Escape

Restore focus on close

Clearly indicate primary action

Avoid unnecessary confirmation dialogs.

---

# Visual Density

Avoid clutter.

Prefer generous spacing.

Reduce visual noise.

Whitespace improves comprehension.

---

# Typography

Use typography for hierarchy.

Not colour.

Limit font sizes.

Limit font weights.

Maintain rhythm.

---

# Icons

Use one icon family.

Maintain consistent stroke width.

Align icons optically.

Avoid mixing styles.

---

# Colour

Colour should communicate.

Examples:

Primary actions

Warnings

Errors

Success

Selection

Avoid decorative colours.

---

# Shadows

Use subtle elevation.

Avoid heavy shadows.

Depth should communicate interaction.

---

# Glass & Blur

Use blur sparingly.

Heavy blur impacts performance.

Apply only where it improves hierarchy.

---

# Accessibility

Every polished interface should also be accessible.

Support:

Keyboard navigation

Reduced motion

Screen readers

High contrast

Large text

Accessibility improves overall quality.

---

# Browser Feel

The browser should feel:

Immediate

Fluid

Reliable

Responsive

Modern

Professional

Users should never notice hesitation.

---

# Quality Checklist

Before completing UI work ask:

Does every button have hover?

Does every input have focus?

Are loading states implemented?

Are empty states useful?

Are animations consistent?

Is spacing consistent?

Do colours match the design language?

Do components align correctly?

Does resizing work?

Does keyboard navigation work?

Would a designer approve this?

---

# Anti Patterns

Never:

Leave unfinished hover states

Animate everything

Mix animation durations

Use inconsistent spacing

Use multiple icon libraries

Use arbitrary border radii

Flash loading indicators

Freeze interfaces

Use blocking modals unnecessarily

Leave empty pages unexplained

Ignore reduced motion

Ignore accessibility

---

# Definition of Premium

Premium software feels:

Predictable

Intentional

Fast

Elegant

Refined

Consistent

Invisible

The user should focus on their task—not the interface.

---

# Gold Standard

Every screen should feel as though it was designed and refined over months by a dedicated product design team.

Interactions should communicate confidence.

Animations should feel natural.

Components should behave consistently.

Nothing should feel unfinished.

The browser should feel closer to products like Arc, Linear, Raycast, Zen Browser, VS Code, and modern native operating system applications than to a typical Electron application.

Users should describe the browser as "smooth", "polished", and "professional" without being able to identify exactly why.
