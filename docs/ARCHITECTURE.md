# Dandelion architecture

Dandelion is a Chromium browser. It is not an embedder, a shell, or an
application built on top of a browser engine — it is Chromium, with Dandelion's
own product identity, user interface and features compiled into it.

## The overlay model

Dandelion does **not** fork `chromium/src`.

A literal fork — copying Chromium's tree and editing it in place — makes every
upstream release a merge of tens of thousands of commits into files you have
modified. Chromium ships a new milestone roughly every four weeks. That model
is what Microsoft Edge does, and it costs a dedicated team to sustain.

Instead, Dandelion follows the approach used by Brave and Vivaldi:

- Chromium is a **pinned dependency**, checked out by `depot_tools` and never
  committed to this repository.
- This repository is **mounted into** that checkout as `src/dandelion`, where
  upstream never writes.
- Dandelion's own code lives entirely under `src/dandelion`, so it cannot
  conflict with an upstream change.
- Upstream files are modified only through a small set of **patches**, each one
  deliberate and reviewable.

The cost of a Chromium roll is therefore proportional to the size of the patch
set, not to the size of Chromium. Keeping that set small is the single most
important long-term constraint on this codebase.

```text
C:\src\chromium\            <- gclient root, not version controlled
├── .gclient                <- generated by build/sync.py
└── src\                    <- chromium/src, pinned to CHROMIUM_VERSION
    ├── chrome\             <- upstream, patched only where unavoidable
    ├── content\
    ├── third_party\
    └── dandelion\  ─────┐  <- junction into this repository
                         │
c:\Users\chris\Desktop\Dandelion\   <- THIS repository
├── CHROMIUM_VERSION        <- the pinned Chromium release
├── app\                    <- product identity: BRANDING, icons, themes
├── browser\                <- browser-process code, including Views UI
├── components\             <- reusable, layered feature components
├── patches\                <- diffs against upstream Chromium files
├── build\                  <- sync, patch and build tooling; GN args
└── docs\
```

## Where code belongs

Dandelion mirrors Chromium's own layering, so that a Chromium engineer can
navigate it without explanation.

| Directory | Contents |
| --- | --- |
| `app/` | Product identity: `BRANDING`, icons, product strings |
| `browser/` | Browser-process logic; `browser/ui/views/` for native UI |
| `components/` | Self-contained features usable by more than one layer |

Rules that follow from Chromium's own architecture:

- All Dandelion C++ lives in `namespace dandelion`.
- `components/` may not depend on `browser/`; the dependency runs one way.
- Per-profile state is a `KeyedService`, registered with a
  `BrowserContextKeyedServiceFactory` — not a singleton or a global.
- User-visible settings are Chromium **prefs**, registered through the standard
  pref registry so that they inherit sync, policy and profile isolation.
- Features are gated behind `base::Feature` flags, so they can ship dark.
- Cross-process communication uses **Mojo**. Nothing else.

## Patching upstream

A patch is permanent maintenance cost. Before writing one, prefer in order:

1. An existing Chromium extension point — a delegate, observer, factory or
   `base::Feature`.
2. A new Dandelion class under `//dandelion` that upstream calls into via a
   single-line hook.
3. Only then, a substantive change to upstream code.

The goal is that most patches are one or two lines: a call into Dandelion code
that itself lives in this repository. See [PATCHES.md](PATCHES.md).

## Rolling Chromium

1. Update `CHROMIUM_VERSION` to the new release tag.
2. `python build/sync.py --no-patch`
3. `python build/apply_patches.py` and rebase whatever conflicts.
4. `python build/update_patches.py --prune` to capture the rebased result and
   drop patches upstream has absorbed.
5. Rebuild and test.

Because Dandelion's own code is untouched by the roll, steps 3 and 4 are the
entire cost — and they scale with the patch count.
